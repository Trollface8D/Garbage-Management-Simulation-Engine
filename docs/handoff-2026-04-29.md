## The mistake to fix first

> "From #2 you mean that to satisfy the policy enforcement you have an
> internal list for generating agents behind the user's back so the user
> will not get what they selected? What is the point of allowing user
> selecting entity in entity list?"

**Yes — that is exactly what the current code does, and it is wrong.**

### Current (broken) data flow

1. The user curates an entity list on the code page:
   - extracts entities from causal artifacts
   - groups them with Gemini
   - manually adds missing ones
   - selects a subset
2. They pick metrics and click **Generate**.
3. The frontend (`code-gen-workspace.tsx`) calls `POST /code_gen/jobs`
   with `selectedEntities = full preview list`.
4. The backend pipeline:
   - **State 1** (`state1_entity_list`) re-extracts entities from the
     causal data via Gemini → produces `entities[]` totally
     independently of the user's curated list.
   - **State 1b** generates a policy outline targeting State 1's entity
     IDs.
   - **State 2** generates `entities/<id>.py` for each State 1 entity.
   - **State 4** generates `policies/<rule>.py` referencing State 1
     entity IDs.

The user-curated entity list **never reaches** State 2 or State 4. The
"selectedEntities" the frontend sends is read from `job.preview.entities`
(which IS State 1's output), so the curation work is silently
discarded for code-gen purposes.

### Why this happened

The pipeline was designed to extract entities from causal data
end-to-end. The page-level entity list / grouping / manual-add UI was
layered on top later for **metric selection** and **wordcloud display**,
but never wired into the code-gen path. The hidden preview entity panel
in the workspace was the last place this leak was visible; hiding it in
commit `110724c` made the disconnect even more confusing instead of
fixing the root cause.

### What "fixing it" looks like

The page-level entity list MUST be the single source of truth.
Concrete options the next conversation should evaluate:

1. **Replace State 1 with the user's list.**
   - `state1_entity_list` becomes a deterministic stage that consumes
     the user's entities (sent in the manifest) and emits the same
     `{id, label, type, frequency}` shape downstream stages already
     expect.
   - State 1b still runs (policy outline derived from causal text)
     but its `target_entity_id` field must be **constrained to
     entities present in the user list** — drop or remap any policy
     whose target isn't in the user list.
   - State 2 / State 4 then just work because the contract is
     unchanged at their layer.

2. **Show the LLM's extra entities as suggestions, not silent additions.**
   - If State 1b proposes a policy targeting an entity the user didn't
     pick, surface that gap in the UI: "Policy `policy_collect`
     references entity `truck` which isn't in your list. Add it?"
   - Don't silently inflate the user's list.

3. **Use the existing entity templates** at
   `Experiment/code_generation/entity_design/entity/gemini_3_pro_entity/template/`
   as the codegen scaffolding instead of generating entity classes
   from scratch. Files present:
     - `entity_object_template.py` (172 lines)
     - `entity_template.py` (13 lines)
     - `environment_template.py` (180 lines)
     - `policy_template.py` (54 lines)
   The current `code_gen_prompts.py` doesn't reference these. The new
   pipeline should fill them in rather than asking Gemini to produce
   class structures from nothing.

The user explicitly called out the templates as unused, so this is part
of the same architectural correction.

### Run-log schema (`metrics.jsonl`, locked)

First line is a header; subsequent lines are samples.

```jsonl
{"_": "header", "run_id": "...", "started_at": "ISO", "tick_seconds": 300.0,
 "metrics": [{"name": "queue_length", "label": "...", "unit": "items",
              "agg": "mean", "viz": "line", "chart_group": null,
              "grounding": "causal_implicit"}, ...]}
{"t": 0.0,   "metric": "queue_length", "entity_id": null, "value": 12.0}
{"t": 300.0, "metric": "queue_length", "entity_id": null, "value": 14.0}
```

`t` is real-world seconds (`tick_number * tick_seconds`) so the log is
self-describing regardless of cadence. `entity_id` is nullable for
environment-level metrics.

## Decisions locked

1. **Tick → real time:** `tick_seconds` lives in the run header.
   Default 300s (5 min). `t` in samples is real seconds.
2. **No LLM at sim runtime.** All generated `*.py` files are pure
   deterministic Python. SLM was considered and explicitly rejected
   ("would create bottleneck process").
3. **Local Windows app target.** No remote runners assumed.
4. **Metric grounding is honest:** every metric carries one of
   `causal_explicit | causal_implicit | domain_inference`.
5. **JSONL is the source of truth** for the run log. PowerBI is a
   secondary BI consumer (deferred).
6. **Insights pipeline is OUT OF SCOPE** here — that's a downstream
   pipeline that consumes this one's logs.
7. **Caching:** explicit context cache for State 2, gated on
   `CODE_GEN_CACHE_MIN_ENTITIES` (default 5),
   `CODE_GEN_CACHE_TTL_SECONDS` (default 1800). Falls back inline
   on any failure.
8. **API contracts use Pydantic models + HTTPException**, not
   `dict[str, Any]` + `JSONResponse`. Frontend `parseError` reads
   both `error` and `detail`. Saved as a memory.
9. **Branch discipline:** one branch + one PR per session
   (`claude/great-antonelli-be4e0a` here). Saved as a memory.

## What's blocked / unresolved

### A. Single source of truth for entities (the core mistake)

- Current State 1 ignores user-curated entity list.
- State 1b's policies target State 1 entity IDs that may not match
  user-known entity names.
- State 2/4 generate code based on State 1's universe, not user's.

**Fix proposal for next session:**
- Pass `userEntityList: [{id, label, type}]` in the `POST /jobs`
  manifest.
- In `_stage_state1_entity_list`, if `userEntityList` is present,
  **skip the Gemini call** and emit the user's list verbatim in the
  expected shape.
- In `_stage_state1b_policy_outline`, after the LLM returns its
  outline, validate every `target_entity_id` against the user list.
  Drop or surface mismatches as warnings.
- Frontend: stop reading `job.preview.entities` for the
  `selectedEntities` payload — send the page-level list directly.

### B. Use the existing entity templates

`Experiment/code_generation/entity_design/entity/gemini_3_pro_entity/template/*.py`
contains hand-authored scaffolds the codegen prompts should fill in
rather than ask the LLM to invent class structure each time. Consider
making these the basis of the State 2 / 3 / 4 prompts so the LLM
fills in the body / methods, not the skeleton.

### C. Worker auto-spawn on POST /jobs (mostly fixed)

- Solved for preview path via `previewOnly: true` (commit `9a05578`).
- Generate path still creates a fresh job and calls `/resume` — works
  but means State 1+1b get re-run in the second job. Once the State 1
  contract is fixed (item A), consider running everything inline /
  via resume on the SAME job to halve State 1+1b cost.

### D. Reporter attribute sampling is best-effort

- `reporter.py` walks `env.entities` and matches by class-name
  substring. If generated entity attribute names don't match
  `metric.required_attrs`, that metric falls silent for the tick.
- Once the templates are in use this should be deterministic; the
  templates fix attribute names.

### E. SQLite dual-write (deferred)

- `fact_simulation_metrics` schema sketched, not built.
- PowerBI report templates not authored.
- User agreed deferring to "after we have measured baseline numbers"
  — same gating principle as caching.
